# 解説モデル比較（sonnet vs opus）Q1〜10

> 同じ問題に対する解説を2モデルで生成。品質・トーン・過不足を比べてください。

## Q1（正解: **A** / 投票 A:95% / B:5%）

**問題**: ある企業が、複数の大陸にまたがる都市の気温・湿度・大気圧データを収集しています。各拠点から毎日収集するデータの平均量は 500 GB で、各拠点には高速インターネット接続があります。
この企業は、すべてのグローバル拠点のデータをできる限り迅速に単一の Amazon S3 バケットに集約したいと考えています。ソリューションは運用上の複雑さを最小限に抑える必要があります。
これらの要件を満たすソリューションはどれですか？

**選択肢**
- A. ✅ 転送先の S3 バケットで S3 Transfer Acceleration を有効にする。マルチパートアップロードを使用して、各拠点のデータを直接転送先 S3 バケットにアップロードする。
- B. 各拠点から最も近いリージョンの S3 バケットにデータをアップロードする。S3 Cross-Region Replication を使用してオブジェクトを転送先 S3 バケットにコピーする。その後、元の S3 バケットからデータを削除する。
- C. AWS Snowball Edge Storage Optimized デバイスのジョブを毎日スケジュールして、各拠点から最も近いリージョンにデータを転送する。S3 Cross-Region Replication を使用してオブジェクトを転送先 S3 バケットにコピーする。
- D. 各拠点から最も近いリージョンの Amazon EC2 インスタンスにデータをアップロードする。Amazon Elastic Block Store (Amazon EBS) ボリュームにデータを保存する。定期的に EBS スナップショットを取得し、転送先 S3 バケットがあるリージョンにコピーする。そのリージョンで EBS ボリュームを復元する。

### 🟦 sonnet

S3 Transfer Acceleration は CloudFront のエッジロケーションを経由してデータを高速転送する機能で、地理的に離れた拠点から S3 へのアップロード速度を大幅に向上させる。各拠点に高速インターネット接続があるため、ネットワーク経由での転送が現実的であり、マルチパートアップロードと組み合わせることで 500 GB のデータも効率的に転送できる。追加インフラが不要で運用の複雑さも最小に抑えられる。B は Cross-Region Replication の設定と元バケットのデータ削除という余分な手順が必要で複雑さが増す。C の Snowball Edge は物理デバイスの配送が伴うため「できる限り早急に」という要件を満たさない。D は EBS スナップショット経由という迂回路が複雑でコスト・管理オーバーヘッドも高い。

### 🟩 opus

正解は A。単一の S3 バケットへ世界中から最速で集約したいケースでは、S3 Transfer Acceleration がエッジロケーション（CloudFront)経由で AWS バックボーンに乗せて長距離転送を高速化し、マルチパートアップロードで 500 GB を並列分割して送れる。追加インフラ不要で運用負荷も最小。
- B: 近隣リージョンへ一旦アップロードし CRR で複製、さらに元を削除する構成は手順が多く遅延も追加され複雑。
- C: Snowball Edge は高速回線がある環境では不要で、物理配送により「できる限り迅速」に反する。
- D: EC2/EBS/スナップショット経由は運用が煩雑で S3 集約に最も遠い。

## Q2（正解: **C** / 投票 C:99% / A:1%）

**問題**: ある企業が、独自開発したアプリケーションのログファイルを分析する必要があります。ログは Amazon S3 バケットに JSON 形式で保存されています。クエリはシンプルでオンデマンドで実行されます。ソリューションアーキテクトは、既存のアーキテクチャへの変更を最小限に抑えて分析を実施する必要があります。
運用上のオーバーヘッドを最小限に抑えながらこれらの要件を満たすには、ソリューションアーキテクトは何をすべきですか？

**選択肢**
- A. Amazon Redshift を使用してすべてのコンテンツを一箇所にロードし、必要に応じて SQL クエリを実行する。
- B. Amazon CloudWatch Logs を使用してログを保存する。Amazon CloudWatch コンソールから必要に応じて SQL クエリを実行する。
- C. ✅ Amazon Athena を Amazon S3 と直接連携させて、必要に応じてクエリを実行する。
- D. AWS Glue を使用してログをカタログ化する。Amazon EMR 上の一時的な Apache Spark クラスターを使用して、必要に応じて SQL クエリを実行する。

### 🟦 sonnet

Amazon Athena は S3 上のデータを直接 SQL クエリで分析できるサーバーレスサービスで、既存の S3 アーキテクチャを変更せずにそのまま活用できる。JSON 形式のデータにも対応しており、オンデマンドのシンプルなクエリというユースケースに最適で、クラスター管理などの運用作業が不要。A の Redshift はデータのロードが必要で既存アーキテクチャへの変更が大きく、大規模 DWH 向けのオーバースペック。B の CloudWatch Logs は S3 からの移行が必要で既存構成の変更が生じ、SQL クエリ機能も限定的。D の Glue + EMR は大規模バッチ処理向けで、オンデマンドのシンプルなクエリには過剰であり運用オーバーヘッドが高い。

### 🟩 opus

正解は C。S3 上の JSON ログに対してオンデマンドでシンプルな SQL を投げるなら、Amazon Athena が最適。サーバーレスでスキャン量課金のみ、S3 を直接クエリでき、既存アーキテクチャを一切変えずに導入できるため運用オーバーヘッドが最小。
- A: Redshift はデータをロードするクラスタ運用が必要で、オンデマンドの軽いクエリには過剰。
- B: CloudWatch Logs は SQL クエリ実行の場ではなく（Insights は独自クエリ言語)、要件に合わない。
- D: Glue + EMR/Spark はクラスタ管理が発生しオーバーヘッドが大きい。

## Q3（正解: **A** / 投票 A:97% / B:3% / C:1%）

**問題**: ある企業が AWS Organizations を使用して、複数の部門にわたる複数の AWS アカウントを管理しています。管理アカウントには、プロジェクトレポートを格納した Amazon S3 バケットがあります。この企業は、この S3 バケットへのアクセスを、AWS Organizations 内の組織に属するアカウントのユーザーのみに制限したいと考えています。
運用上のオーバーヘッドを最小限に抑えながらこれらの要件を満たすソリューションはどれですか？

**選択肢**
- A. ✅ S3 バケットポリシーに、組織 ID を参照する aws:PrincipalOrgID グローバル条件キーを追加する。
- B. 各部門に組織単位 (OU) を作成する。S3 バケットポリシーに aws:PrincipalOrgPaths グローバル条件キーを追加する。
- C. AWS CloudTrail を使用して、CreateAccount・InviteAccountToOrganization・LeaveOrganization・RemoveAccountFromOrganization イベントを監視する。それに応じて S3 バケットポリシーを更新する。
- D. S3 バケットへのアクセスが必要な各ユーザーにタグを付ける。S3 バケットポリシーに aws:PrincipalTag グローバル条件キーを追加する。

### 🟦 sonnet

aws:PrincipalOrgID 条件キーを S3 バケットポリシーに設定することで、指定した Organizations の組織 ID に属するすべてのアカウントのプリンシパルからのアクセスだけを許可できる。新たなアカウントが組織に追加されても自動的に適用されるため、ポリシーの変更は不要で運用オーバーヘッドが最小。B の aws:PrincipalOrgPaths は OU 単位の制御が必要な場合に使うが、「組織全体」を対象とする要件には冗長で、OU の作成・管理が別途必要になる。C は CloudTrail でアカウント変更を検知して手動でバケットポリシーを更新する方式で、タイムラグと運用コストが大きい。D のタグベース制御はユーザーごとにタグを付与・管理する作業が必要で、スケールしない。

### 🟩 opus

正解は A。組織内の全アカウントのプリンシパルにのみアクセスを絞るには、S3 バケットポリシーの Condition に aws:PrincipalOrgID を指定し、対象の Organizations ID と一致させるのが最もシンプル。アカウントの増減があっても組織 ID は不変なのでポリシー更新が不要。
- B: PrincipalOrgPaths は特定 OU 単位の制御向けで、OU 作成という余計な作業が増える。
- C: CloudTrail でイベント監視しポリシーを都度更新するのは運用負荷が高く、リアルタイム性もない。
- D: 個別タグ付けは全ユーザー管理が必要で運用が煩雑。

## Q4（正解: **A** / 投票 A:99% / D:1%）

**問題**: あるアプリケーションが VPC 内の Amazon EC2 インスタンス上で動作しています。このアプリケーションは Amazon S3 バケットに保存されたログを処理します。EC2 インスタンスはインターネットへの接続なしに S3 バケットにアクセスする必要があります。
Amazon S3 へのプライベートネットワーク接続を提供するソリューションはどれですか？

**選択肢**
- A. ✅ S3 バケットへのゲートウェイ VPC エンドポイントを作成する。
- B. ログを Amazon CloudWatch Logs にストリームする。ログを S3 バケットにエクスポートする。
- C. Amazon EC2 にインスタンスプロファイルを作成して S3 へのアクセスを許可する。
- D. Amazon API Gateway API をプライベートリンクで作成して S3 エンドポイントにアクセスする。

### 🟦 sonnet

ゲートウェイ VPC エンドポイントを作成すると、VPC 内の EC2 インスタンスがインターネットを経由せず AWS のプライベートネットワーク経由で S3 にアクセスできる。追加コストなしで設定でき、ルートテーブルにエントリを追加するだけで有効化される最もシンプルな解決策。B は CloudWatch Logs への転送という構成変更が必要で、要件である「S3 へのプライベート接続」を直接解決しない。C のインスタンスプロファイルは IAM の認証・認可の仕組みであり、ネットワーク経路（インターネット経由か否か）を制御するものではない。D の API Gateway + PrivateLink は S3 アクセス用として過剰に複雑な構成で、標準的な解決策ではない。

### 🟩 opus

正解は A。VPC 内から S3 へインターネットを経由せずプライベート接続するには、S3 用のゲートウェイ VPC エンドポイントを作成する。ルートテーブルにエンドポイント向けの経路が入り、追加料金なしで S3(および DynamoDB)にプライベート到達できる。
- B: CloudWatch Logs 経由は接続経路の話ではなく要件を満たさない。
- C: インスタンスプロファイル/IAM は認可の仕組みで、ネットワーク経路(プライベート接続)は提供しない。
- D: API Gateway で S3 をプライベート化する構成は不自然で、標準解ではない。

## Q5（正解: **C** / 投票 C:98% / D:2%）

**問題**: ある企業が、ユーザーがアップロードしたドキュメントを Amazon EBS ボリュームに保存する単一の Amazon EC2 インスタンスを使用して、AWS 上で Web アプリケーションをホストしています。可用性とスケーラビリティを向上させるために、アーキテクチャを複製し、別のアベイラビリティーゾーンに 2 台目の EC2 インスタンスと EBS ボリュームを作成し、両方を Application Load Balancer の背後に配置しました。この変更後、ユーザーから「ウェブサイトを更新するたびに、ドキュメントの一部しか表示されず、すべてのドキュメントを同時に確認できない」との報告がありました。
ユーザーがすべてのドキュメントを同時に参照できるようにするために、ソリューションアーキテクトはどのような解決策を提案すべきですか？

**選択肢**
- A. 両方の EBS ボリュームにすべてのドキュメントが含まれるようにデータをコピーする。
- B. Application Load Balancer を設定して、ユーザーをドキュメントが存在するサーバーに誘導する。
- C. ✅ 両方の EBS ボリュームのデータを Amazon EFS にコピーする。新しいドキュメントを Amazon EFS に保存するようにアプリケーションを変更する。
- D. Application Load Balancer を設定して、リクエストを両方のサーバーに送信する。各ドキュメントを正しいサーバーから返す。

### 🟦 sonnet

EBS ボリュームは単一の EC2 インスタンスにのみアタッチされるため、複数インスタンス間でファイルを共有できず、ALB がリクエストを振り分けるたびに異なるインスタンスのデータしか見えない問題が生じる。Amazon EFS（Elastic File System）は複数の EC2 インスタンスから同時にマウント可能な共有ファイルシステムであり、すべてのインスタンスが同じドキュメントにアクセスできるようになる。A は手動コピーで一時的に揃えられても新規アップロード時に再び不整合が生じる。B の ALB によるルーティングはドキュメントの所在をアプリ側で追跡する複雑な仕組みが必要で根本解決にならない。D も同様にアプリケーション側での管理が必要で、スケールアウト時の運用が困難。

### 🟩 opus

正解は C。各 EC2 がそれぞれの EBS にファイルを持つため、ALB で振り分けられた先のインスタンスのファイルしか見えないのが原因。共有ファイルシステムである Amazon EFS に既存データを集約し、以後の保存先も EFS にすれば全インスタンスから同じドキュメント一式を参照できる。EBS は単一 AZ・単一インスタンスにしかアタッチできない点が本質的な制約。
- A: 手動コピーは新規追加分が同期されず破綻する。
- B/D: ALB はどのファイルがどのサーバーにあるかを判断できず、ルーティングでは解決しない。

## Q6（正解: **B** / 投票 B:88% / C:11% / D:1%）

**問題**: ある企業が、オンプレミスのネットワーク接続ストレージに NFS を使用して大容量の動画ファイルを保存しています。各動画ファイルのサイズは 1 MB から 500 GB の範囲です。総ストレージ容量は 70 TB で、増加は止まっています。この企業は動画ファイルを Amazon S3 に移行することを決定しました。使用するネットワーク帯域幅をできる限り抑えつつ、できる限り早急に動画ファイルを移行する必要があります。
これらの要件を満たすソリューションはどれですか？

**選択肢**
- A. S3 バケットを作成する。S3 バケットへの書き込み権限を持つ IAM ロールを作成する。AWS CLI を使用してすべてのファイルをローカルから S3 バケットにコピーする。
- B. ✅ AWS Snowball Edge ジョブを作成する。オンプレミスで Snowball Edge デバイスを受け取る。Snowball Edge クライアントを使用してデータをデバイスに転送する。AWS が Amazon S3 にデータをインポートできるようにデバイスを返送する。
- C. オンプレミスに S3 File Gateway をデプロイする。S3 File Gateway に接続するためのパブリックサービスエンドポイントを作成する。S3 バケットを作成する。S3 File Gateway 上に新しい NFS ファイル共有を作成する。新しいファイル共有を S3 バケットに向ける。既存の NFS ファイル共有から S3 File Gateway にデータを転送する。
- D. オンプレミスネットワークと AWS の間に AWS Direct Connect 接続をセットアップする。オンプレミスに S3 File Gateway をデプロイする。S3 File Gateway に接続するためのパブリック仮想インターフェイス (VIF) を作成する。S3 バケットを作成する。S3 File Gateway 上に新しい NFS ファイル共有を作成する。新しいファイル共有を S3 バケットに向ける。既存の NFS ファイル共有から S3 File Gateway にデータを転送する。

### 🟦 sonnet

70 TB のデータをネットワーク経由で転送すると帯域幅の消費が膨大になる。AWS Snowball Edge は物理デバイスにデータをロードして AWS に送付することで、インターネット帯域を消費せずに大容量データを移行できる。増加が止まっている静的なデータセットであるため、デバイス配送の時間を許容すれば最速かつ最もネットワーク効率が良い手段。A の AWS CLI によるコピーは 70 TB 分の帯域を大量消費し、転送時間も長くなる。C の S3 File Gateway はネットワーク経由での転送になるため帯域の節約にならず、一回限りの移行には設定が過剰。D は Direct Connect の開通に数週間かかるうえコストも高く、「できる限り早急に」という要件に反する。

### 🟩 opus

正解は B。70 TB を「帯域幅を極力使わず」かつ「できる限り早く」移行するなら、物理デバイスにデータを載せて送る AWS Snowball Edge が定石。オンライン転送だと大量帯域と長時間を要するが、Snowball ならネットワーク負荷ゼロに近く大容量を短期で S3 に取り込める。
- A: AWS CLI での直接コピーは 70 TB では帯域を大量消費し時間もかかる。
- C/D: S3 File Gateway は継続的なハイブリッドアクセス向けで、一括大量移行では回線を使い時間もかかる。D の Direct Connect は敷設に時間・コストがかかりさらに不利。

## Q7（正解: **D** / 投票 D:86% / A:11% / C:2% / B:1%）

**問題**: ある企業が、受信メッセージを取り込むアプリケーションを保有しています。その後、数十もの他のアプリケーションやマイクロサービスがこれらのメッセージを迅速に消費します。メッセージ数は大幅に変動し、突然 1 秒あたり 100,000 件に急増することもあります。この企業はソリューションを疎結合化し、スケーラビリティを向上させたいと考えています。
これらの要件を満たすソリューションはどれですか？

**選択肢**
- A. メッセージを Amazon Kinesis Data Analytics に永続化する。コンシューマーアプリケーションがメッセージを読み取って処理するように設定する。
- B. 取り込みアプリケーションを Auto Scaling グループ内の Amazon EC2 インスタンスにデプロイし、CPU メトリクスに基づいて EC2 インスタンス数をスケーリングする。
- C. 単一のシャードを持つ Amazon Kinesis Data Streams にメッセージを書き込む。AWS Lambda 関数を使用してメッセージを前処理し、Amazon DynamoDB に保存する。コンシューマーアプリケーションが DynamoDB からメッセージを読み取って処理するように設定する。
- D. ✅ 複数の Amazon Simple Queue Service (Amazon SQS) サブスクリプションを持つ Amazon Simple Notification Service (Amazon SNS) トピックにメッセージをパブリッシュする。コンシューマーアプリケーションがキューからメッセージを処理するように設定する。

### 🟦 sonnet

SNS トピックに対して複数の SQS キューをサブスクリプションとして設定するファンアウトパターンは、1 つのメッセージを数十のコンシューマーへ独立して届けるのに最適。SQS がバッファとして機能するため、急激なスパイク（秒間 100,000 件）にも対応でき、プロデューサーとコンシューマーを疎結合に保てる。A の Kinesis Data Analytics はストリームのリアルタイム分析サービスであり、メッセージのファンアウト配信には適さない。B の EC2 Auto Scaling だけではコンシューマーへの疎結合なメッセージ配信の仕組みを提供しない。C の単一シャード Kinesis は秒間 1,000 件・1 MB のスループット上限があり、秒間 100,000 件のスパイクに対応できない。

### 🟩 opus

正解は D。数十のコンシューマーが同じメッセージを受け取り、秒間 10 万件の急増にも耐える疎結合構成には SNS のファンアウトが最適。SNS トピックに publish し、各コンシューマー用の SQS キューをサブスクライブさせれば、各サービスが独立してバッファリング・スケールしながら消費でき、取り込み側と処理側が完全に分離される。
- A: Kinesis Data Analytics は永続化ストアではなくストリーム分析用で用途違い。
- B: EC2/CPU ベースの Auto Scaling は疎結合化にならず急増への追従も遅い。
- C: 単一シャードは 10 万件/秒に到底耐えられずボトルネックになる。

## Q8（正解: **B** / 投票 B:93% / D:4% / C:2% / A:2%）

**問題**: ある企業が分散アプリケーションを AWS に移行しています。このアプリケーションは可変ワークロードに対応しています。既存のプラットフォームは、複数のコンピュートノードにわたってジョブを調整するプライマリサーバーで構成されています。この企業は、回復性とスケーラビリティを最大化するソリューションでアプリケーションをモダナイズしたいと考えています。
これらの要件を満たすために、ソリューションアーキテクトはアーキテクチャをどのように設計すべきですか？

**選択肢**
- A. Amazon Simple Queue Service (Amazon SQS) キューをジョブの送信先として設定する。Auto Scaling グループで管理される Amazon EC2 インスタンスでコンピュートノードを実装する。スケジュールベースのスケーリングを使用するように EC2 Auto Scaling を設定する。
- B. ✅ Amazon Simple Queue Service (Amazon SQS) キューをジョブの送信先として設定する。Auto Scaling グループで管理される Amazon EC2 インスタンスでコンピュートノードを実装する。キューのサイズに基づいて EC2 Auto Scaling を設定する。
- C. Auto Scaling グループで管理される Amazon EC2 インスタンスでプライマリサーバーとコンピュートノードを実装する。AWS CloudTrail をジョブの送信先として設定する。プライマリサーバーの負荷に基づいて EC2 Auto Scaling を設定する。
- D. Auto Scaling グループで管理される Amazon EC2 インスタンスでプライマリサーバーとコンピュートノードを実装する。Amazon EventBridge (Amazon CloudWatch Events) をジョブの送信先として設定する。コンピュートノードの負荷に基づいて EC2 Auto Scaling を設定する。

### 🟦 sonnet

SQS キューをジョブのバッファとして使うことでプライマリサーバーの役割をキューに置き換え、単一障害点をなくせる。キューのサイズ（ApproximateNumberOfMessages）に基づいた Auto Scaling により、ジョブの滞留量に応じてコンピュートノードが自動的にスケールアウト・インするため、可変ワークロードへの対応と回復性の最大化を両立できる。A はスケジュールベースのスケーリングであり、予測不能な可変ワークロードには不適切。C の CloudTrail はジョブキューとしての機能を持たない監査ログサービスで用途が全く異なる。D の EventBridge はイベント駆動のルーティングサービスであり、ジョブの滞留バッファとしての役割を果たさないため、コンピュートノードのスケーリング判断基準としても不適切。

### 🟩 opus

正解は B。プライマリが多数のコンピュートノードにジョブを分配する構成は、SQS キューにジョブを積み、ワーカーを Auto Scaling グループの EC2 で処理する疎結合モデルへ置き換えると回復性・スケーラビリティが最大化する。スケーリングの契機はキューの深さ(ApproximateNumberOfMessages)に基づくのがベストで、実需要に正確に追従できる。
- A: スケジュールベースは可変ワークロードに合わない。
- C: CloudTrail は監査ログでありジョブの送信先にはならない。
- D: プライマリサーバーを残す構成は単一障害点を温存し、EventBridge もジョブキューの代替にはならない。

## Q9（正解: **B** / 投票 B:84% / D:12% / A:3%）

**問題**: ある企業が、データセンターで SMB ファイルサーバーを運用しています。このファイルサーバーには、作成後最初の数日間は頻繁にアクセスされる大容量ファイルが保存されています。7 日後はほとんどアクセスされなくなります。
総データ容量が増加し続けており、会社の総ストレージ容量に近づいています。ソリューションアーキテクトは、最近アクセスされたファイルへの低レイテンシーアクセスを維持しながら、会社の利用可能なストレージ容量を増やす必要があります。また、将来のストレージ問題を回避するためのファイルライフサイクル管理も提供する必要があります。
これらの要件を満たすソリューションはどれですか？

**選択肢**
- A. AWS DataSync を使用して、SMB ファイルサーバーから 7 日以上経過したデータを AWS にコピーする。
- B. ✅ Amazon S3 File Gateway を作成して会社のストレージ容量を拡張する。S3 ライフサイクルポリシーを作成して、7 日後にデータを S3 Glacier Deep Archive に移行する。
- C. Amazon FSx for Windows File Server ファイルシステムを作成して会社のストレージ容量を拡張する。
- D. 各ユーザーのコンピューターに Amazon S3 にアクセスするためのユーティリティをインストールする。S3 ライフサイクルポリシーを作成して、7 日後にデータを S3 Glacier Flexible Retrieval に移行する。

### 🟦 sonnet

S3 File Gateway をオンプレミスに配置することで、既存の NFS インターフェイスを維持しながら S3 をバックエンドストレージとして利用でき、ローカルキャッシュによって最近アクセスされたファイルへの低レイテンシーアクセスも確保される。S3 ライフサイクルポリシーで 7 日後に S3 Glacier Deep Archive へ自動移行することで、コスト効率の良いライフサイクル管理も実現できる。A の DataSync は一方向のデータコピーツールで、継続的なキャパシティ拡張やリアルタイムアクセスには対応していない。C の FSx for Windows File Server は SMB 専用でスケーラブルではあるが、S3 との自動ライフサイクル連携機能を持たず、コストも高い。D はユーザー端末へのユーティリティインストールが必要で運用負荷が高く、既存の NFS 環境を活かせない。

### 🟩 opus

正解は B。オンプレのストレージ枯渇を回避しつつ、最近のファイルへ低レイテンシーでアクセスさせるには Amazon S3 File Gateway が適する。ローカルキャッシュで直近アクセスを高速化しながら実体は S3 に置き、容量を実質無制限に拡張できる。さらに S3 ライフサイクルポリシーで 7 日後に Glacier Deep Archive へ移行すれば、アクセスされなくなった古いファイルを自動でコスト最適化しつつライフサイクル管理を実現できる。
- A: DataSync は一括/定期コピーで、キャッシュによる低レイテンシー継続アクセスやライフサイクル管理は提供しない。
- C: FSx for Windows は容量拡張にはなるが自動アーカイブのライフサイクル管理がない。
- D: 各端末へのユーティリティ導入は SMB 共有の置き換えにならず運用も煩雑。

## Q10（正解: **B** / 投票 B:98% / A:2%）

**問題**: ある企業が AWS 上で eコマース Web アプリケーションを構築しています。このアプリケーションは新規注文に関する情報を処理のために Amazon API Gateway REST API に送信します。この企業は、注文が受け付けられた順番どおりに処理されることを保証したいと考えています。
これらの要件を満たすソリューションはどれですか？

**選択肢**
- A. API Gateway 統合を使用して、アプリケーションが注文を受信したときに Amazon Simple Notification Service (Amazon SNS) トピックにメッセージをパブリッシュする。処理を実行するために AWS Lambda 関数をトピックにサブスクライブする。
- B. ✅ API Gateway 統合を使用して、アプリケーションが注文を受信したときに Amazon Simple Queue Service (Amazon SQS) FIFO キューにメッセージを送信する。SQS FIFO キューが処理のために AWS Lambda 関数を呼び出すように設定する。
- C. API Gateway オーソライザーを使用して、アプリケーションが注文を処理している間はすべてのリクエストをブロックする。
- D. API Gateway 統合を使用して、アプリケーションが注文を受信したときに Amazon Simple Queue Service (Amazon SQS) 標準キューにメッセージを送信する。SQS 標準キューが処理のために AWS Lambda 関数を呼び出すように設定する。

### 🟦 sonnet

SQS FIFO キューは名前のとおり先入れ先出しの順序保証を提供し、同一メッセージグループ内でメッセージが受信した順番どおりに処理されることを保証する。API Gateway から SQS FIFO キューに直接統合し、Lambda をイベントソースとして設定することで、シンプルかつ順序保証のある注文処理パイプラインを構築できる。A の SNS はメッセージの順序保証を提供しないため、注文順序の維持には不適切。C のオーソライザーによるリクエストブロックは処理中の新規注文をすべて拒否する手法で、ユーザー体験を著しく損なう。D の SQS 標準キューはベストエフォートの順序でしかなく、重複配信の可能性もあるため、順序保証が必要な注文処理には使用できない。

### 🟩 opus

正解は B。注文を受付順に厳密に処理するには、順序保証と重複排除を備える SQS FIFO キューを使う。API Gateway から FIFO キューへ送り、Lambda で消費すれば FIFO 内で先入れ先出しが担保される。
- A: SNS は順序保証がなく（FIFO トピックはあるが本選択肢は標準トピック想定)、複数配信で順序が崩れる。
- C: オーソライザーは認証認可の仕組みで順序制御には無関係。
- D: SQS 標準キューはベストエフォート順序かつ少なくとも一度配信で、受付順の処理を保証できない。
