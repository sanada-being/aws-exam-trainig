# 解説サンプル（opus・本番マージ後）先頭6問

## Q1（正解: **A**）

**問題**: ある企業が、複数の大陸にまたがる都市の気温・湿度・大気圧データを収集しています。各拠点から毎日収集するデータの平均量は 500 GB で、各拠点には高速インターネット接続があります。
この企業は、すべてのグローバル拠点のデータをできる限り迅速に単一の Amazon S3 バケットに集約したいと考えています。ソリューションは運用上の複雑さを最小限に抑える必要があります。
これらの要件を満たすソリューションはどれですか？

**解説**

正解は A。S3 Transfer Acceleration は CloudFront のエッジロケーションを経由して最適化された AWS バックボーンにデータを載せるため、遠隔地からの長距離アップロードを高速化でき、マルチパートアップロードと組み合わせれば大容量転送も並列化できて運用も簡単。
- B: 一度近隣リージョンに置いてから CRR で複製し元を削除するのは往復と削除処理が増え、遅延も運用複雑さも大きい。
- C: Snowball Edge は物理デバイスの往復に日数がかかり、高速回線があり毎日転送する用途には遅く不適。
- D: EC2/EBS を経由しスナップショットをコピー・復元する構成は手数が多く、S3 集約という目的に対して過剰かつ複雑。
要点: Transfer Acceleration はグローバル拠点からの単一バケット集約の定番。

## Q2（正解: **C**）

**問題**: ある企業が、独自開発したアプリケーションのログファイルを分析する必要があります。ログは Amazon S3 バケットに JSON 形式で保存されています。クエリはシンプルでオンデマンドで実行されます。ソリューションアーキテクトは、既存のアーキテクチャへの変更を最小限に抑えて分析を実施する必要があります。
運用上のオーバーヘッドを最小限に抑えながらこれらの要件を満たすには、ソリューションアーキテクトは何をすべきですか？

**解説**

正解は C。Amazon Athena は S3 上の JSON/CSV/Parquet をサーバーレスでそのままクエリでき、事前のデータロードやインフラ構築が不要なので、既存構成をほぼ変えずオンデマンド SQL 分析を実現できる。
- A: Redshift は事前にデータをロードしクラスターを常時運用する必要があり、単純なオンデマンドクエリには過剰。
- B: CloudWatch Logs は既に S3 にあるログの保存先ではなく、標準 SQL クエリも実行できない。
- D: Glue + EMR/Spark はクラスター管理が発生し運用オーバーヘッドが大きい。
要点: S3 データの手軽な SQL 分析なら Athena が第一候補。

## Q3（正解: **A**）

**問題**: ある企業が AWS Organizations を使用して、複数の部門にわたる複数の AWS アカウントを管理しています。管理アカウントには、プロジェクトレポートを格納した Amazon S3 バケットがあります。この企業は、この S3 バケットへのアクセスを、AWS Organizations 内の組織に属するアカウントのユーザーのみに制限したいと考えています。
運用上のオーバーヘッドを最小限に抑えながらこれらの要件を満たすソリューションはどれですか？

**解説**

正解は A。aws:PrincipalOrgID 条件キーを S3 バケットポリシーに付ければ、指定した組織 ID に属するすべてのアカウントを一括で許可でき、アカウントの増減にもポリシー変更なしで追従できる。
- B: aws:PrincipalOrgPaths は特定 OU 配下に絞る用途で、組織全体を対象にする本要件には不要に複雑。
- C: CloudTrail でアカウント変更を監視しポリシーを都度更新するのは運用負荷が高く自動化としても脆い。
- D: 各ユーザーへのタグ付けと PrincipalTag 管理は手作業が増え、組織単位の制御には非効率。
要点: 組織全体へのアクセス制御は aws:PrincipalOrgID が最小オーバーヘッド。

## Q4（正解: **A**）

**問題**: あるアプリケーションが VPC 内の Amazon EC2 インスタンス上で動作しています。このアプリケーションは Amazon S3 バケットに保存されたログを処理します。EC2 インスタンスはインターネットへの接続なしに S3 バケットにアクセスする必要があります。
Amazon S3 へのプライベートネットワーク接続を提供するソリューションはどれですか？

**解説**

正解は A。S3 はゲートウェイ VPC エンドポイントに対応しており、ルートテーブル経由でインターネットを介さずプライベートに S3 へアクセスできる（追加料金もなし）。
- B: CloudWatch Logs 経由のエクスポートはログ転送の話で、S3 へのプライベート接続経路を提供しない。
- C: インスタンスプロファイル(IAM)はアクセス権限を与えるだけでネットワーク経路を私設化しない。
- D: S3 はゲートウェイエンドポイントで直接接続でき、API Gateway/PrivateLink を挟むのは不要で誤り。
要点: S3・DynamoDB はゲートウェイ型、他の多くのサービスはインターフェイス型 VPC エンドポイント。

## Q5（正解: **C**）

**問題**: ある企業が、ユーザーがアップロードしたドキュメントを Amazon EBS ボリュームに保存する単一の Amazon EC2 インスタンスを使用して、AWS 上で Web アプリケーションをホストしています。可用性とスケーラビリティを向上させるために、アーキテクチャを複製し、別のアベイラビリティーゾーンに 2 台目の EC2 インスタンスと EBS ボリュームを作成し、両方を Application Load Balancer の背後に配置しました。この変更後、ユーザーから「ウェブサイトを更新するたびに、ドキュメントの一部しか表示されず、すべてのドキュメントを同時に確認できない」との報告がありました。
ユーザーがすべてのドキュメントを同時に参照できるようにするために、ソリューションアーキテクトはどのような解決策を提案すべきですか？

**解説**

正解は C。各 EC2 が別々の EBS を持つとドキュメントが分散する。共有ファイルシステムである Amazon EFS に集約し、複数 AZ の全インスタンスからマウントすれば、どのサーバーに振られても全ドキュメントを参照できる。
- A: 一度コピーしても新規アップロードは片側にしか書かれず、すぐ再び不整合になる。
- B/D: ALB でドキュメントの所在に応じて振り分けるのは非現実的で、ロードバランサーの役割ではない。
要点: EBS は単一インスタンス専用、複数インスタンス共有には EFS(NFS)を使う。

## Q6（正解: **B**）

**問題**: ある企業が、オンプレミスのネットワーク接続ストレージに NFS を使用して大容量の動画ファイルを保存しています。各動画ファイルのサイズは 1 MB から 500 GB の範囲です。総ストレージ容量は 70 TB で、増加は止まっています。この企業は動画ファイルを Amazon S3 に移行することを決定しました。使用するネットワーク帯域幅をできる限り抑えつつ、できる限り早急に動画ファイルを移行する必要があります。
これらの要件を満たすソリューションはどれですか？

**解説**

正解は B。70 TB を帯域を極力使わず短期間で移行するには、物理デバイスにデータを載せて送る AWS Snowball Edge が最適。ネットワーク転送を回避しつつ大容量を迅速に S3 へ取り込める。
- A: AWS CLI での直接コピーは 70 TB 分のネットワーク帯域を大量消費し時間もかかる。
- C: S3 File Gateway 経由はインターネット回線でのアップロードとなり帯域を使い遅い。
- D: Direct Connect の敷設は時間とコストがかかり、迅速性の要件に合わない。
要点: 数十TB以上を回線を使わず移行するオフライン手段が Snowball Edge。
