# 公開・広告収益化の技術アーキテクチャ

最終更新: 2026-07-01

前提（オーナー方針）:
- リポジトリ/サイトは公開のまま進める。
- 問題コンテンツは現行のまま（翻訳は自作の草の根サイトとして運用）。
- **運営者の個人情報を公開しないこと**を必須条件とする。
- ※AdSenseは審査側判断でコンテンツ性質により通らない可能性がある点は技術前提として認識する（本書は技術実現に限定）。

関連issue: #36(適法化) / #37(匿名化) / #38(配信) / #39(AdSense) / #40(CMP・法務) / #41(集客・計測)

## 全体構成

```
GitHub(repo) ──build──▶ Cloudflare Pages ──▶ 独自/中立ドメイン ──▶ ユーザー
                              静的SPA(Vite+React)+PWA
                                     │
        ┌────────────┬───────────────┼───────────────┬───────────┐
     AdSense       CMP(同意)      ルーティング      解析        性能最適化
   (広告ユニット) (Cookie同意)   (法定ページ)  (Cookieレス)  (CWV/分割)
```

## 1. ホスティングと匿名URL（#38 / #37）
- **Cloudflare Pages** に GitHub 連携デプロイ。プロジェクト名を中立名 → `xxx.pages.dev`（GitHubユーザー名が出ない）。
- 独自ドメインは **Cloudflare Registrar** 管理＝**WHOIS代理公開がデフォルト**で登録者情報を非露出。屋号/ハンドルで登録。
- `vite base` を `'/'` に変更（github.io のサブパス `/aws-exam-training/` を廃止）。環境でビルド分岐。
- BrowserRouter 用に Cloudflare Pages の **SPAフォールバック**（`_redirects` に `/* /index.html 200`）を設定。
- 露出面の残課題: **公開repoのコミット履歴に実名・会社メールが残る**。サイトURLがrepoを指さなくなれば導線は切れるが、完全匿名化には「匿名アカウントへ履歴なしで再作成」＋今後は**匿名 git identity（ハンドル＋noreplyメール）**でのコミットが必要。→ 要方針決定。
- AdSense/ドメインの実名・納税情報は各社にのみ提供され公開されない。公開面（URL/WHOIS/コミット導線）を匿名化する。

## 2. AdSense 組み込み（#39）
- `adsbygoogle.js`（`data-ad-client=ca-pub-xxxx`）を読み込み、広告枠を **React コンポーネント化**（`AdUnit.tsx`：`useEffect` で `(adsbygoogle=window.adsbygoogle||[]).push({})`）。
- **SPA特有の注意**:
  - 画面はstate遷移のため、同一スロットを二重 push しない。画面/表示ごとに `key` を分けて確実に再マウント。
  - **CLS対策**として広告枠の高さを予約（`min-height`）。
  - 広告スクリプトは**同意取得後に読み込む**（CMP連動）。
- **配置ポリシー**: 選択肢付近は誤クリック誘発でポリシー違反 → **ホーム下部・結果画面・解説の下**など非侵襲な位置に限定。1画面あたりの枠数も過剰にしない。

## 3. 同意管理 CMP（#40）
- EEA/UK 向けに **Google認定CMP必須**。最小実装は **AdSense標準「プライバシーとメッセージ」（旧Funding Choices）**＝管理画面で作成すれば同意バナーが自動表示、追加コードほぼ不要。
- 同意状態に応じ パーソナライズ/非パーソナライズ広告を切替。同意前は広告Cookieを置かない。

## 4. ルーティング＆法定ページ（#40）
- 現状は画面をstateで切替＝独立URLページが無い。AdSense審査/SEOで**独立したプライバシーポリシーURL**等が必要なため **react-router-dom を導入**。
- 追加ページ:
  - `/privacy` プライバシーポリシー（広告Cookie・解析・第三者送信の明示）
  - `/disclaimer` 免責（学習補助であり合否/正確性を保証しない、AWS/Amazon商標は各社帰属）
  - `/about` 運営者（屋号/ハンドル）
  - `/contact` 問い合わせ（フォーム or 専用メール。個人非特定）
- Cloudflare Pages の SPA フォールバックで BrowserRouter の直リンクに対応。

## 5. 解析（#41）
- **Plausible / Umami**（Cookieレス・軽量・同意負荷小）を推奨。GA4を使う場合は同意連動必須。
- 流入・収益の可視化。

## 6. 性能・収益最適化（#41）
- 現状 **データJSON 3.3MB を起動時一括ロード**。公開＆広告サイトでは Core Web Vitals が評価/収益に影響 → **分割ロード（トピック/範囲単位）・遅延読み込み・コード分割**。
- 広告スクリプト/外部タグは **PWAキャッシュ対象外**にする（SW設定）。

## 実装順（推奨）
1. #38 配信基盤（Cloudflare Pages＋中立/独自ドメイン＋`base:'/'`＋SPAフォールバック）
2. ルーティング導入＋法定ページ雛形（#40の技術基盤）
3. CMP＋AdSense枠（#40同意 → #39広告）
4. 解析・性能最適化（#41）

## リスク/留意（技術前提）
- コンテンツ性質によりAdSense審査に通らない可能性（#36）。技術実装とは別に審査側判断。
- 匿名性は「公開面の非露出」で担保。既存コミット履歴の実名は別途対応が必要（#37）。
